# 产品定位

项目代号叫：

> **Water Me**

Slogan：

> **Humans need watering, too.**
>
> 人类，也需要浇水。

整个产品理念：

> 根据真实工作状态，在最合适的时候提醒喝水、站立、休息，而不是机械地按照时钟提醒。

------

# MVP（V1）

目标只有一句话：

> **让用户安装后，不需要任何学习，就能开始使用。**

因此：

**不开复杂设置。**

**没有桌宠。**

**没有统计。**

**没有账号。**

先把最重要的一条链路跑通。

------

# MVP 功能模块

整个系统其实只有六个模块。

```
Water Me
│
├── Activity Monitor
│
├── Reminder Engine
│
├── Notification Window
│
├── Scheduler
│
├── Settings
│
└── Tray
```

------

# 模块一：Activity Monitor（活动监测）

这是整个软件最重要的模块。

职责：

监控：

- 鼠标
- 键盘
- 锁屏
- 睡眠
- 唤醒

输出统一状态：

```
Idle

Working

Locked

Sleeping
```

例如：

```
Working

↓

累计工作时间

25min
```

如果：

```
Idle

超过5分钟
```

那么：

暂停计时。

而不是：

继续倒计时。

这一点和很多提醒软件最大的区别就在这里。

------

# 模块二：Reminder Engine（提醒引擎）

这个模块只有一件事：

决定：

什么时候提醒。

例如：

```
Working

↓

60min

↓

Water
```

或者：

```
Working

↓

45min

↓

Stand Up
```

后续：

可以继续增加：

```
Eye Break

Walk

Lunch

Medicine

Stretch
```

提醒类型全部统一。

例如：

```rust
enum ReminderType {
    Water,
    Stand,
    EyeBreak,
}
```

以后几乎不用改架构。

------

# 模块三：Notification Window

这是 MVP 唯一需要展示给用户看的窗口。

建议不要直接：

Windows Notification。

而是：

自己做一个：

Overlay。

例如：

```
┌─────────────────────┐

💧

Water Me!!

You've been working
for 60 minutes.

[Done]

[Snooze]

└─────────────────────┘
```

整个窗口：

- 永远置顶
- 带动画
- 半透明
- 自动关闭

体验会比系统通知好很多。

------

# 模块四：Scheduler

实际上就是：

Timer。

但是：

它不是：

```
Clock Timer
```

而是：

```
Working Timer
```

例如：

```
开始工作

↓

15min

↓

离开

↓

暂停

↓

回来

↓

继续

↓

45min

↓

提醒
```

这也是整个软件的核心。

------

# 模块五：Settings

MVP：

设置越少越好。

建议：

```
喝水间隔

60min

────────────

站立间隔

45min

────────────

闲置多久暂停

5min

────────────

开机启动

ON

────────────

全屏提醒

ON
```

不要超过：

6 个设置。

------

# 模块六：Tray

系统托盘即可。

右键：

```
Water Me

────────────

暂停提醒

恢复提醒

立即喝水

设置

退出
```

不用主窗口。

------

# MVP 工作流程

整个流程其实非常简单。

```
启动

↓

监听用户活动

↓

开始累计工作时间

↓

达到提醒阈值

↓

弹出提醒

↓

↓

```

结束。

# 为什么不要桌宠？

桌宠其实属于：

表现层（Presentation）。

而不是：

业务层（Business）。

真正的架构应该是：

```
Business

↓

Reminder

↓

Notification

↓

Desktop Pet（以后）
```

以后：

Notification

完全可以替换成：

```
Notification

↓

Desktop Pet

↓

Overlay

↓

Windows Notification

↓

Mobile Push
```

业务逻辑完全不用动。

# V6（跨平台生态）

当 Windows 版本稳定后，可以逐步扩展：

- **macOS**：适配系统状态监听、通知中心和菜单栏应用。
- **Linux**：支持主流桌面环境（GNOME、KDE）的托盘与通知。
- **移动端联动（远期）**：电脑离开后，将提醒同步到手机，而不是继续在电脑端等待。

# 我建议的开发路线（按优先级）

| 版本     | 目标               | 开发重点                             | 预计代码占比 |
| -------- | ------------------ | ------------------------------------ | ------------ |
| **V1**   | 可用的健康提醒工具 | 活动监测 + 提醒引擎 + Overlay + 托盘 | **60%**      |
| **V1.5** | 提升体验           | 配置持久化、动画、开机启动、日志     | **10%**      |
| **V2**   | 数据价值           | 每日统计、历史记录、连续打卡         | **10%**      |
| **V3**   | 产品特色           | 桌宠模式、情绪状态、角色系统         | **15%**      |
| **V4**   | 用户留存           | 成长系统、成就、主题与皮肤           | **5%**       |

## 一个额外建议：从第一天就按“引擎 + UI”解耦

结合你之前做 **Tauri 2** 项目的经验，我建议采用下面这条边界：

```
┌──────────────────────────────┐
│ Reminder Engine（纯 Rust）    │
│ • 活动监测                    │
│ • 工作时间统计                │
│ • 提醒规则                    │
│ • 状态管理                    │
└──────────────┬───────────────┘
               │ Event
               ▼
┌──────────────────────────────┐
│ UI Layer（Tauri + React）     │
│ • Overlay 提醒                │
│ • 设置页面                    │
│ • 托盘                        │
│ • （未来）桌宠                │
└──────────────────────────────┘
```

这样未来把 Overlay 换成桌宠、增加统计页，甚至适配其他平台时，都不需要改动核心提醒逻辑。对于一个长期演进的项目，这是最值得提前投入的架构决策。



# 提示

这个问题其实是**这个产品最核心的问题**。

如果解决不了它，你做得再漂亮，也只是另一个「用了三天就卸载」的提醒软件。

------

## 为什么提醒会失效？

因为**人不是因为忘记喝水，而是主动选择忽略提醒。**

想象一下：

```
💧 该喝水了

[关闭]
```

你的大脑会瞬间做一个决策：

> "我知道了，等会儿。"

然后：

30 分钟过去了。

1 小时过去了。

最后一天都没喝。

所以问题从来不是：

> 如何提醒？

而是：

> 如何让用户愿意响应提醒？

------

## 市面上很多软件的问题

它们都是：

```
Timer

↓

到时间

↓

Notification

↓

结束
```

整个流程：

```
提醒

↓

用户关闭

↓

软件认为任务完成
```

其实：

没有完成。

用户只是：

**关闭了窗口。**

------

## 你真正应该记录的是

不是：

```
提醒次数
```

而是：

```
提醒

↓

用户有没有完成
```

例如：

```
60 分钟

↓

提醒

↓

Done（喝了）

✓
```

或者：

```
提醒

↓

Later
```

或者：

```
提醒

↓

Ignore
```

三个动作。

意义完全不同。

------

# MVP 我建议增加三个按钮

不要只有：

```
×

关闭
```

而是：

```
💧 Water Me

────────────

[我喝了]

[10分钟后提醒]

[今天跳过]
```

这样：

Reminder Engine 就知道：

这是：

完成。

还是：

延期。

还是：

放弃。

------

# 不要惩罚用户

很多健康软件喜欢：

```
连续三天没喝水

❌

失败
```

用户：

直接卸载。

因为：

没人喜欢被教育。

------

我反而建议：

一种更温柔的方式。

例如：

```
🥺

今天一直没有喝水。

希望你现在已经喝了一杯。
```

结束。

没有：

"你失败了。"

------

# 我觉得可以加入一点人格

例如：

第一次：

```
🌱

Water Me.
```

第二次：

```
🌱

Still waiting...
```

第三次：

```
🥺

I'm getting thirsty.
```

第四次：

```
😭

Please...
```

第五次：

```
💀

...
```

这就是桌宠以后存在的意义。

不是：

越来越烦。

而是：

越来越有情绪。

------

# 更重要的一点

不要提醒：

"喝水"

而是：

提醒：

"站起来。"

例如：

```
已经连续工作

90 分钟。

去倒杯水吧。

顺便活动一下。
```

实际上：

你的目标不是：

喝水。

而是：

让用户离开电脑。

只要：

站起来。

去接水。

走几十步。

眼睛离开屏幕。

其实：

三个目标全部完成。

所以：

提醒内容可以变成：

> 去给自己接一杯水。

而不是：

> 请喝水。

这是两个完全不同的心理感受。

------

# 我还有一个比较大胆的想法

你的产品理念是：

> **Humans need watering, too.**

那么提醒其实可以反过来。

不是：

> "你该喝水了。"

而是：

```
🌱

I think you forgot me.
```

或者：

```
🌱

Could you water me?
```

甚至：

```
🌱

I'll wait here.
```

因为：

真正需要浇水的是：

用户自己。

但是：

他说的是：

植物。

这种拟人化会让提醒没有那么强的命令感。

------

## 我认为真正值得研究的，不是提醒，而是**行为设计**

你可以把整个流程设计成这样：

```
工作

↓

提醒出现

↓

不是要求"喝水"

↓

而是邀请用户
"去接一杯水，活动一下身体"

↓

用户点击"我回来了"

↓

记录一次健康习惯
```

这样软件记录的是**健康行为**，而不是一个机械计时器。

**最后，我建议把这个产品的目标从「提醒软件」改成「帮助建立健康习惯的软件」。**

这两个定位看起来只差几个字，但产品设计会完全不同：

- **提醒软件**关注的是「通知有没有弹出来」。
- **习惯软件**关注的是「用户有没有真正完成这个行为」。

后者才更有长期价值，也更符合你后续想加入桌宠、成长系统和游戏化设计的发展方向。
